# 奄美飲（ノ）ミュニティマップ 仕様書（ドラフト）

> 既存コード（2026-07-01時点）からの逆算に加え、サービスの背景・目的について認識合わせした内容を反映しています。未確定の仕様は「要確認」としてマークしています。

## 1. サービス概要

- **サービス名**: 奄美飲（ノ）ミュニティマップ
- **キャッチコピー**: 奄美群島ゆかりの居酒屋を探そう
- **対象地域（店舗の所在地）**: 日本各地（店主・関係者のルーツは奄美群島＝奄美大島・喜界島・徳之島・沖永良部島・与論島・加計呂麻島／与路島／請島）

### 背景・課題
奄美群島の島々と、本土で暮らす人たち（進学・就職等で島を離れた出身者や、その先の世代、ゆかりのない一般の人々）との関係性が年々希薄化していることが課題。

### コンセプトと仮説
一方で、奄美群島にルーツを持つ店主が営む居酒屋・食堂・バーなどの飲食店が、「小さなコミュニティ」として機能している例が現在進行形で各地に多数存在する。ルーツを持たない一般客であっても、常連として通ううちに店主の出身の島に興味を持ち、実際に足を運んでみる、といった**関係人口の創出**がすでに自然発生的に起きている。

このサービスは、そうした全国に散らばる「飲ミュニティ（飲む＋コミュニティ）」の存在を**見える化**し、**アクセスしやすくする**ことで、この関係人口創出の流れを後押しすることを目的とする。

### 目指す成果（KPI・ゴールの方向性、要すり合わせ）
- 奄美群島にルーツを持つ飲食店の認知・来店促進
- ルーツを持たない一般ユーザーが「飲ミュニティ」を通じて奄美群島に関心を持ち、実際に訪島するきっかけを作る（関係人口の創出・拡大）
- 出身者同士・出身者と非出身者のつながりの可視化

## 2. 対象ユーザー

1. **奄美群島にルーツを持つ人**: 島を離れて本土で暮らす出身者・その子孫。故郷の味や同郷の人とのつながりを求めている
2. **ルーツを持たない一般ユーザー（潜在的関係人口）**: たまたま「飲ミュニティ」を知り、常連になったり、店主の出身の島に興味を持ち実際に訪れたりする可能性がある層。サービスとして最も拡大を狙いたい対象
3. **店舗の店主・情報提供者**: 自身の店を掲載したい、または常連客・知人としてその店の情報を教えたい人（広告掲載＝`isAd`枠、情報提供フォーム・`providerNote`と対応）

## 3. 画面構成

### 3.1 全体レイアウト（`src/app/page.tsx`）
- ヘッダー（`Header`）＋検索・絞り込みバー（`SearchFilters`）＋メインエリア
- メインエリアはマップ（`MapView`）とリスト（`StoreList`）の2ペイン構成
  - **PC（sm以上）**: マップ＋右サイドに幅96（w-96）のリストを並列表示
  - **モバイル（sm未満）**: マップ／リストをヘッダー右上のトグルボタンで切替表示（どちらか一方のみ表示）
- 店舗選択時、`StoreDetailModal` を画面下部（モバイル）／中央（PC）にモーダル表示

### 3.2 ヘッダー（`Header.tsx`）
- サービス名・キャッチコピー表示
- モバイル時のみ「マップ／リスト」切替トグルを表示

### 3.3 検索・絞り込み（`SearchFilters.tsx`）
- キーワード検索（店名・住所・町・集落・ジャンルを対象に部分一致、大文字小文字区別なし）
- 島フィルター（複数選択可、トグル式）— 奄美大島／喜界島／徳之島／沖永良部島／与論島
- ジャンルフィルター（複数選択可、トグル式）— `STORES` データから動的生成
- 絞り込みはAND条件（島 AND ジャンル AND キーワード）、それぞれ複数選択時はOR

### 3.4 マップビュー（`MapView.tsx`）
- **地図プロバイダ**: Google Maps（`@vis.gl/react-google-maps`）
- 初期表示: 中心 `{lat: 35.3, lng: 137.5}`（日本列島中心付近）、ズーム6（要確認：奄美群島ではなく日本全体が見える設定。店舗が全国に散らばっている前提と一致）
- 店舗ごとにピン（`AdvancedMarker` + `Pin`）を表示。ピンの色は出身島ごとに色分け（`ISLANDS`定義）
- 選択中の店舗はピンが1.2倍に拡大
- ピンクリックで吹き出し（`InfoWindow`）内に `StoreBalloon`（サムネ・店名・ジャンル・「詳細を見る」ボタン）を表示
- 地図の空白部分クリックで選択解除
- APIキー未設定時はエラーメッセージを表示（地図を描画しない）

### 3.5 リストビュー（`StoreList.tsx` / `StoreCard.tsx` / `StoreSummary.tsx`）
- 絞り込み後の件数表示（「n件のお店」）
- 0件時は「条件に一致するお店が見つかりませんでした。」を表示
- 各カードはサムネ画像・島バッジ・ジャンル・店名・住所を表示。クリックで選択状態にし、同時に詳細モーダルを開く

### 3.6 店舗詳細モーダル（`StoreDetailModal.tsx`）
- 写真ギャラリー（横スクロール）
- 店名・ジャンル・住所・町/集落・電話番号（あれば）・プロフィール文
- 店舗ページへのリンク（`storeUrl`があれば）
- 予約ボタン（`isAd`かつ`reservationUrl`がある場合のみ表示）
- **広告枠（`isAd: true`）による表示差**（意図的な広告課金モデル。現段階では基本システム構築を優先し、課金・申込みフローは未実装。データ構造・表示差分のみ先行して準備している段階）
  - 写真: 通常5枚まで／広告枠は20枚まで
  - プロフィール文字数: 通常400文字まで／広告枠は1500文字まで
  - ヘッダーに「PR」バッジを表示

## 4. データモデル（`src/types/store.ts`）

```ts
type IslandKey = "amami" | "kikai" | "tokunoshima" | "okinoerabu" | "yoron";

interface Store {
  id: string;
  name: string;            // 店名
  genre: string;           // ジャンル（自由記述、フィルターの選択肢は実データから動的生成）
  island: IslandKey;       // ゆかりの島
  prefecture: string;      // 店舗所在の都道府県
  town: string;            // 店舗所在の市区町村
  village: string;         // 店主の出身集落・町（例:「笠利町出身」）
  address: string;         // 住所
  lat: number;
  lng: number;
  photos: string[];        // 画像URL配列
  profile: string;         // 紹介文
  storeUrl?: string;       // 店舗外部ページ
  phone?: string;
  isAd: boolean;           // 広告掲載フラグ
  reservationUrl?: string; // 予約リンク（広告枠のみ使用）
  providerNote?: string;   // 情報提供者に関するメモ（運営内部用、非公開想定）
}

interface StoreFilters {
  keyword: string;
  islands: IslandKey[];
  genres: string[];
}
```

- 現状は `src/data/stores.ts` にハードコードされたサンプルデータ12件のみ。今後はDB（Supabase等）に移行予定（第7章参照）
- 写真は現状すべて Unsplash のダミー画像URL

## 5. 島マスタ（`src/data/islands.ts`）

各島ごとに以下を定義：表示名、ピン色（`color`）、薄色背景（`colorSoft`）、枠線色（`colorBorder`）、テキスト色（`textColor`）。バッジやフィルターボタンの配色に使用。

## 6. 非機能・技術要件

- **フレームワーク**: Next.js（App Router）+ TypeScript + Tailwind CSS
- **地図**: Google Maps JavaScript API（`NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` が必須。未設定時はフォールバックUIを表示）
- 認証機能: 無し
- API Route / バックエンド: 無し（フロントエンドのみ、静的データ）
- 画像最適化: `next/image` 使用だが `unoptimized` 指定（外部ドメインの最適化設定を回避）

## 7. 決定事項（2026-07-01 認識合わせ）

1. **店舗データの管理方法**: ハードコードをやめ、簡易DB（例: Supabase）＋管理画面を用意する。基本システムのスコープに含める。
2. **広告枠（`isAd`）の運用**: 意図的な広告課金モデル。ただし課金・申込みフローは今回のスコープ外。データ構造・表示差分（写真枚数／文字数上限／PRバッジ）のみ先行実装し、将来の運用に備える。
   - マネタイズの内訳（想定）:
     a. **広告費**: 基本の登録情報（無料枠）に加えて費用を払うことで、写真枚数・プロフィール文字数を拡張した「レポート記事」的な紹介文（現状の`profile`欄1500文字上限をそのままレポート記事として扱う。独立したコンテンツ枠は設けない）と、予約導線（`reservationUrl`のボタン表示）を掲載できる
     b. **予約代行手数料**（将来検討・今回はスコープ外）: 運営が店舗への予約を仲介し、手数料を得るモデル。現時点ではDB構造・機能としては用意せず、将来の収益源の一つとして本仕様書に概念のみ記録する
3. **情報提供者メモ（`providerNote`）**: 店舗詳細画面に一般公開する（例:「情報提供: 〇〇さん」の形で表示）。
4. **店舗情報の投稿導線**: フルの投稿・編集機能は作らない。店主・情報提供者が「この店舗の情報を教える」ための簡単な情報提供フォームのみ用意する。実際のデータ反映（編集・公開）は運営が管理画面から行う。
5. **ユーザーアカウント機能**: お気に入り・レビュー投稿・ログインは今回のスコープに含めない。閲覧専用。
6. **地図初期表示**: ユーザーの現在地情報を取得し、近くの店舗が見やすい中心・ズームに寄せる（位置情報取得不可・拒否時は現行のデフォルト中心`{lat: 35.3, lng: 137.5}`・ズーム6にフォールバック）。
7. **ジャンルの分類方法**: 自由記述をやめ、マスタ管理にする（管理画面側で事前定義したジャンル一覧から選択入力する形にする）。
8. **公開範囲**: 一般公開（誰でもアクセス・閲覧可能なサービスとする）。
9. **多言語対応**: 日本語のみ。将来必要になった場合に改めて検討。
10. **対象地域の拡張性**: 当面は奄美群島（6カテゴリ: 奄美大島・喜界島・徳之島・沖永良部島・与論島・加計呂麻島／与路島／請島）のみを扱うが、将来的に他の離島・過疎地域など別の「ルーツ」にも対象を広げる可能性を見込み、`IslandKey`のハードコード列挙型ではなく、DB上は「地域／島マスタ」テーブルとして設計する（追加が容易な構造にする）。加計呂麻島・与路島・請島は3島まとめて1カテゴリとして扱う。

11. **計測・分析**: 簡易なアクセス解析ツール（例: Google Analytics等）を導入する。
12. **情報提供フォームの送信先・反映方法**: 送信内容はDBに「未承認」ステータスで保存する。運営が管理画面で内容を確認・承認して初めて公開データに反映される（承認フローあり）。
13. **管理画面の認証方法**: Supabase Authでログインする。運営メンバーのみアカウントを発行し、管理画面はログイン必須にする。

## 8. 今後の実装スコープまとめ

基本システム構築フェーズで着手する内容:
- Supabase等のDB導入・スキーマ設計（`Store`型 + `providerNote` + 承認ステータス管理 + ジャンルマスタ + 地域／島マスタ〔将来の他地域拡張を見込んだ構造〕）

- ユーザーアカウント（お気に入り・レビュー投稿等）
- 多言語対応
